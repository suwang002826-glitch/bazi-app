# 排盘规则

这个文件记录八字排盘采用和待确认的规则。任何排盘算法变化，都先更新这里，再修改代码和验收样例。

当前规则版本：`bazi-rule-profile@2026.07.02-control`

当前阶段目标：以精准、稳定、专业的八字排盘为主线。小程序端只做录入、展示和验证；排盘算法、农历 / 闰月 data-pack、节气、真太阳时和权威来源由后端统一负责。

## 总控口径

以下规则从 2026-07-02 起作为项目总控口径。旧 `v0.1` 内容只保留为历史记录，不能再作为默认算法依据。

1. 算法权威归后端
   小程序不得继续硬写八字算法、节气白名单、农历转换白名单。前端只能调用后端或读取明确标记为预览 / 验收的 data-pack。

2. 权威来源优先级
   紫金山天文台 / 国家授时中心公开资料优先；香港天文台等官方资料用于交叉校验；用户截图、第三方网站、手工表格只能进入候选或交叉校验层，未复核前不得进入稳定运行时。

3. 年柱
   年柱按立春精确交节时刻切年。立春前仍属上一年柱，立春后切到新年柱；不能只按农历春节或公历日期粗略判断。

4. 月柱
   月柱按十二节切月，不按农历月切月。十二节为小寒、立春、惊蛰、清明、立夏、芒种、小暑、立秋、白露、寒露、立冬、大雪，边界以精确交节时刻为准。

5. 日柱与子时
   子时规则必须显式进入规则配置和验收样例。正式算法不得继续默认沿用旧 `v0.1` 的“子时不换日”口径；如支持早晚子时，应作为明确流派选项，并在结果中显示本次采用口径。

6. 真太阳时
   真太阳时必须由后端统一计算，并在输入参数和结果页中明确显示本次是否启用、使用经度和校准后时间。验收样例必须显式写明开 / 关，不允许隐式假设。

7. 农历与闰月
   农历和闰月只负责输入转换与结果展示。内部四柱计算统一使用转换后的公历时间、节气表和规则配置。data-pack 覆盖不足时必须明确降级或拒绝，不能偷用白名单补结果。

8. 结果可信度
   任何候选 data-pack、用户截图样本、第三方表格，都必须标记 `not-runtime-approved` 或同等状态。只有完成来源登记、抽取、复核、差异记录和准入结论后，才能进入稳定运行时。

## 待确认规则

以下规则必须逐项确认，不能只凭默认算法定案：

1. 节气换月  
   月柱是否严格按节气切换，节气交接时刻如何处理。

2. 子时换日  
   23:00-23:59 是否按次日计算，是否区分早子时、晚子时。

3. 真太阳时  
   是否默认启用，出生地经度如何换算，城市经度数据来源是什么。

4. 农历和闰月  
   农历输入是否支持闰月，闰月转换如何校验。

5. 大运起运  
   起运顺逆、起运岁数、起运天数折算采用哪套规则。

6. 流年流月  
   流月是否按节气切换，流日流时边界如何处理。

7. 神煞和格局  
   神煞库、格局判定和喜用神规则的来源与优先级。

## 历史 v0.1 暂定口径

以下口径用于早期 `feature/bazi-accuracy` 的第一轮开发和验收，现仅作历史记录。遇到本节与“总控口径”冲突时，以“总控口径”为准。

### 1. 总原则

- 第一版只做八字排盘准确性。
- 八字准确度优先级高于 AI 解读和 UI 美化。
- 规则集中写入规则配置，不散落在页面事件里。
- 先用验收样例证明差异，再修算法。
- 当前代码结果与规则口径冲突时，先记录差异，再开 PR 修正。

### 2. 年柱

- 第一版按立春切年。
- 不按春节切年。
- 立春交节前仍属上一年柱，立春交节后切换到新年柱。
- 交节边界以精确时间为准，不能只按日期粗略判断。

### 3. 月柱

- 月柱按节气换月。
- 以节气精确交接时刻作为月柱切换边界。
- 第一版先优先校验立春、惊蛰、清明等节气边界样例。

### 4. 日柱与子时

- 旧 `v0.1` 曾默认子时不换日，按北京时间自然日计算日柱。
- 这个口径只能保留为历史兼容样例，不能再作为正式默认。
- 后续必须同时准备“默认口径”和“早晚子时流派选项”的验收样例。

### 5. 时柱

- 时柱按十二时辰计算。
- 默认使用北京时间。
- 如果开启真太阳时，则先把输入时间换算后再判定时辰。

### 6. 真太阳时

- 真太阳时作为可配置项保留，但必须在输入、接口和结果中显式记录。
- 开启时必须记录出生地经度来源，例如城市预设、手动输入或地图选择。
- 真太阳时影响跨时辰、跨日期结果时，必须在结果页显示校准说明。

### 7. 公历、农历和闰月

- 第一版必须支持公历输入。
- 第一版需要支持农历输入和农历闰月输入。
- 农历与闰月转换必须通过验收样例后，才能标记为稳定。
- 农历转公历后的内部计算仍进入统一四柱计算入口。
- 当前第三轮已把农历转换记录移入 `data-pack` 基础版，避免继续在 `calendarAdapter` 中堆代码白名单。
- 当前 `data-pack` 状态仍是 `acceptance-seed`，不代表完整农历库已完成。

### 8. 大运、流年、流月

- 第一版先保留结构和展示位置。
- 起运岁数、顺逆排、大运、流年、流月必须等样例覆盖后再修正。
- 未通过样例前，不对大运起运结果做确定性承诺。

### 9. 神煞、格局、喜用神

- 第一版作为辅助信息展示。
- 优先级低于四柱准确性。
- 神煞库、格局和喜用神来源未完全确认前，标记为“测试版参考”。

### 10. 开发准入

进入正式算法开发前，至少满足：

- #4 已记录本规则口径。
- #5 已记录第一批验收样例。
- `docs/验收样例.md` 中至少包含普通公历、立春前后、子时、农历、闰月样例。
- `feature/bazi-accuracy` 分支只改规则层、校验层和必要算法，不夹带 UI 大改。

## 记录原则

- 对规则不确定时，先写“待确认”，不要直接改代码。
- 每个关键规则至少准备 3 个验收样例。
- 多个来源结果不一致时，记录来源差异。
- 规则负责人确认后，再进入开发任务。
