# 排盘规则

当前规则版本：`v0.1`

当前阶段目标：先确认八字排盘准确性，不扩展六爻、奇门，不顺手重做界面。

## 待确认规则

以下规则必须逐项确认，不能只凭默认算法定案：

1. 节气换月  
   月柱是否严格按节气切换，节气交接时刻如何处理。

2. 子时换日  
   23:00-23:59 是否按次日计算，是否区分早子时、晚子时。

3. 真太阳时  
   是否默认启用，出生地经度如何换算，城市经度数据来源是什么。

4. 农历和闰月  
   农历输入是否支持闰月，闰月转换如何校验。

5. 大运起运  
   起运顺逆、起运岁数、起运天数折算采用哪套规则。

6. 流年流月  
   流月是否按节气切换，流日流时边界如何处理。

7. 神煞和格局  
   神煞库、格局判定和喜用神规则的来源与优先级。

## v0.1 暂定口径

以下口径用于 `feature/bazi-accuracy` 的第一轮开发和验收。遇到不同流派或不同工具结果不一致时，先在 Issue 中记录差异，不直接改死到代码里。

### 1. 总原则

- 第一版只做八字排盘准确性。
- 八字准确度优先级高于 AI 解读和 UI 美化。
- 规则集中写入规则配置，不散落在页面事件里。
- 先用验收样例证明差异，再修算法。
- 当前代码结果与规则口径冲突时，先记录差异，再开 PR 修正。

### 2. 年柱

- 第一版按立春切年。
- 不按春节切年。
- 立春交节前仍属上一年柱，立春交节后切换到新年柱。
- 交节边界以精确时间为准，不能只按日期粗略判断。

### 3. 月柱

- 月柱按节气换月。
- 以节气精确交接时刻作为月柱切换边界。
- 第一版先优先校验立春、惊蛰、清明等节气边界样例。

### 4. 日柱与子时

- 第一版默认子时不换日，按北京时间自然日计算日柱。
- 23:00-23:59 暂不自动按次日处理。
- 后续如果要支持“晚子时换日”，单独做成流派选项，不与默认口径混用。

### 5. 时柱

- 时柱按十二时辰计算。
- 默认使用北京时间。
- 如果开启真太阳时，则先把输入时间换算后再判定时辰。

### 6. 真太阳时

- 第一版做成可选项，默认关闭。
- 开启时必须记录出生地经度来源，例如城市预设、手动输入或地图选择。
- 真太阳时影响跨时辰、跨日期结果时，必须在结果页显示校准说明。

### 7. 公历、农历和闰月

- 第一版必须支持公历输入。
- 第一版需要支持农历输入和农历闰月输入。
- 农历与闰月转换必须通过验收样例后，才能标记为稳定。
- 农历转公历后的内部计算仍进入统一四柱计算入口。
- 当前第三轮已把农历转换记录移入 `data-pack` 基础版，避免继续在 `calendarAdapter` 中堆代码白名单。
- 当前 `data-pack` 状态仍是 `acceptance-seed`，不代表完整农历库已完成。

### 8. 大运、流年、流月

- 第一版先保留结构和展示位置。
- 起运岁数、顺逆排、大运、流年、流月必须等样例覆盖后再修正。
- 未通过样例前，不对大运起运结果做确定性承诺。

### 9. 神煞、格局、喜用神

- 第一版作为辅助信息展示。
- 优先级低于四柱准确性。
- 神煞库、格局和喜用神来源未完全确认前，标记为“测试版参考”。

### 10. 开发准入

进入正式算法开发前，至少满足：

- #4 已记录本规则口径。
- #5 已记录第一批验收样例。
- `docs/验收样例.md` 中至少包含普通公历、立春前后、子时、农历、闰月样例。
- `feature/bazi-accuracy` 分支只改规则层、校验层和必要算法，不夹带 UI 大改。

## 记录原则

- 对规则不确定时，先写“待确认”，不要直接改代码。
- 每个关键规则至少准备 3 个验收样例。
- 多个来源结果不一致时，记录来源差异。
- 规则负责人确认后，再进入开发任务。
