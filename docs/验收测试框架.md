# 八字验收测试框架

## 目标

用固定样例验证八字排盘准确性，避免凭感觉改算法。第一阶段先人工整理样例，后续再接入自动校验。

## 验收流程

```text
产品与验收负责人提供样例
技术实现负责人录入样例结构
运行当前排盘逻辑
比对期望结果和当前结果
记录差异
只修复已确认规则范围内的问题
```

## 样例字段

```js
const acceptanceCase = {
  id: 'BZI-001',
  title: '普通公历生日样例',
  source: '待补充',
  input: {
    mode: 'solar',
    gender: 'female',
    birthDate: '1990-02-04',
    birthTime: '10:30',
    birthPlace: '北京',
    longitude: 116.4074,
    useTrueSolarTime: false
  },
  expected: {
    yearPillar: '待补充',
    monthPillar: '待补充',
    dayPillar: '待补充',
    hourPillar: '待补充',
    fortuneStart: '待补充'
  },
  actual: {
    yearPillar: null,
    monthPillar: null,
    dayPillar: null,
    hourPillar: null,
    fortuneStart: null
  },
  result: 'pending',
  notes: ''
}
```

## 第一批必须覆盖

1. 普通公历生日样例
2. 立春前后样例
3. 节气交接前后样例
4. 子时样例
5. 农历或闰月样例

## 结果状态

```text
pending：等待录入或等待规则确认
pass：当前结果与期望一致
fail：当前结果与期望不一致
blocked：规则未定，暂时不能判断
```

## 差异记录格式

```text
样例编号：BZI-001
输入：1990-02-04 10:30，北京，女，公历
期望：年柱 X，月柱 X，日柱 X，时柱 X
当前：年柱 Y，月柱 Y，日柱 Y，时柱 Y
差异：月柱不一致
可能原因：节气换月边界未按精确交节时间处理
处理建议：等待 #4 明确节气换月规则后修正
```

## 技术负责人当前可做

- 建立样例结构
- 建立比对报告格式
- 标记规则阻塞点
- 准备后续自动校验入口

## 当前自动校验入口

当前已建立第一批验收脚本：

```bash
node scripts/validate-bazi-acceptance.js
```

脚本会校验：

- 第一批 6 个八字基础样例的四柱结果。
- BZI-005、BZI-006 的农历转换公历日期、data-pack 版本和转换来源。
- 农历 data-pack 的 manifest 注册摘要。
- data-pack 外农历输入会明确报错，不会静默按公历或默认日期计算。
- 真太阳时是否默认关闭。
- 八字输入页是否提供真太阳时开关。

注意：

- BZI-005、BZI-006 已升级为真实农历/闰月输入校验，脚本会检查转换出的公历日期、data-pack 版本、转换来源和四柱。
- 当前 data-pack 仍是 `acceptance-seed`，完整农历库和权威数据包仍需后续补齐。
- 包外农历输入当前期望错误码为 `LUNAR_DATE_OUTSIDE_DATA_PACK_COVERAGE`。
- BZI-004 子时样例按 `v0.1` 不换日口径，辛日子时为 `戊子`。

## 技术负责人当前不做

- 不自行补“正确答案”
- 不凭单个网站结果改算法
- 不在 #4 和 #5 未确认前重写核心算法
