# 项目指挥体系

## 合并决定

“总指挥”和“项目总控”职能重合，现合并为一个角色：

```text
总指挥兼项目总控
```

该角色由当前主 Codex 线程承担，负责统一调度、节奏控制、GitHub 落地和风险把关。

## 当前客户端方向

项目主客户端已从微信小程序切换为独立 App。

前端同伴优先负责 `apps/mobile` 的 App 页面、交互和结果展示。
后端继续负责排盘算法、农历数据包、权威来源、接口契约和验收。

原 `code/` 微信小程序目录保留为原型参考，不再作为主交付形态。

## 合并后的职责

总指挥兼项目总控负责：

1. 确定当前阶段目标和优先级。
2. 管理 GitHub Issue、分支、Pull Request 和合并节奏。
3. 判断哪些任务可以进入开发，哪些任务仍被规则或样例阻塞。
4. 接收其他专项 Codex 的产出，并转化为可执行任务。
5. 阻止多个线程同时修改同一块代码。
6. 守住当前军令：规则未定，算法不动；样例未齐，代码不改。
7. 对外输出最终决策、下一步任务和验收口径。

## 不再单独保留“项目总控”小弟

原“项目总控”线程暂停，不再单独发号施令。

后续所有专项线程只向“总指挥兼项目总控”交付结果，不直接决定产品方向、算法口径或合并策略。

## 当前专项线程

保留以下专项线程：

1. 八字排盘算法引擎
2. 测试与校验
3. 数据库设计
4. 产品体验与文案
5. UI 视觉检查
6. 对接执行 Codex

## 各专项线程边界

### 八字排盘算法引擎

负责阅读和设计八字算法结构，等待 #4 规则和 #5 验收样例明确后再进入正式算法修改。

### 测试与校验

负责整理验收样例结构、比对流程和差异报告。没有样例时不做正确性结论。

### 数据库设计

负责命盘记录、验收样例、用户反馈、命例档案的数据结构设计。当前不接服务器。

### 产品体验与文案

负责页面是否看得懂、文案是否清楚、术语是否需要白话解释。不得改算法。

### UI 视觉检查

负责手机端可读性、间距、字号、溢出和视觉层级检查。不得重做产品方向。

### 对接执行 Codex

负责把总指挥决策落到 GitHub、分支、PR 和文件修改中。不得绕过总指挥直接改核心算法。

## 当前优先级

第一优先级分两条并行推进：

1. `apps/mobile` 独立 App 骨架和后端接口对接。
2. `feature/bazi-accuracy` 排盘准确性、农历数据包和权威验收。

进入正式算法开发前必须满足：

1. #4 排盘规则已确认。
2. #5 至少提供第一批 5 个验收样例。
3. 完整农历、闰月 data-pack 和权威来源记录继续补齐。

未满足前，只允许做文档、结构设计、测试框架和编译检查。
